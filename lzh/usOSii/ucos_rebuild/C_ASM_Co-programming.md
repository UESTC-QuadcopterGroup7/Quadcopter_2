# C语言与汇编语言的协同编程机制

在嵌入式开发（如 uC/OS-II 的移植）中，经常需要 C 语言和汇编语言混合编程。C 语言负责上层业务逻辑，而汇编语言则用于处理底层硬件相关的操作（如上下文切换、开关中断等）。为了让这两种语言能够互相“认识”并协同工作，编译器和链接器遵循了一套标准的机制。

---

## 1. 核心原理：链接器（Linker）的符号解析

无论是 C 语言编译出的 `.o`（目标文件），还是汇编语言编译出的 `.o`，最终都要交由**链接器**组合成一个可执行文件（如 `.axf` 或 `.elf`）。
在链接器眼中，没有“函数”或“变量”的概念，只有**“符号”（Symbol）**和**“地址”（Address）**。

当 C 语言调用一个函数时，它实际上是在生成一条跳转到某个“符号”的指令；当汇编语言定义一个函数时，它实际上是在定义这个“符号”对应的代码起始地址。只要两边的“符号名”完全一致，链接器就能把它们连在一起。

---

## 2. 汇编中声明的函数如何在 C 中被调用？

要让 C 语言成功调用汇编里写的函数，需要经过以下三个关键步骤：

### 第一步：在汇编中“导出”（Export）符号
汇编语言默认所有的标签（Label）都是私有的（局部符号），外部文件是看不见的。必须使用特定的伪指令将符号暴露给全局。
在 ARM 汇编（Keil/ARMCC）中，使用 `EXPORT` 关键字：

```assembly
    AREA |.text|, CODE, READONLY, ALIGN = 2
    THUMB

    ; 导出函数名，让外部 C 文件能够看到这个符号
    EXPORT  OS_CPU_SR_Save

OS_CPU_SR_Save
    MRS     R0, PRIMASK      ; 读取中断屏蔽寄存器
    CPSID   I                ; 关闭中断
    BX      LR               ; 返回（LR 中保存了返回地址）
```

### 第二步：在 C 语言头文件中“声明”（Declare）函数
C 语言编译器在编译时（此时还没到链接阶段）是非常严格的。当它在 `.c` 文件中遇到一个函数调用时，它必须知道这个函数的**参数类型**和**返回值类型**，否则就会报错或产生警告。
因此，必须在 `.h` 文件中声明这个函数，告诉 C 编译器：“这个函数存在，它的长相是这样的，具体的实现你别管，链接器以后会帮你找到的。”

```c
// 在 os_cpu.h 中声明汇编函数
// 告诉 C 编译器：这是一个没有参数，返回值为 OS_CPU_SR 类型的函数
OS_CPU_SR OS_CPU_SR_Save(void);
```

### 第三步：在 C 语言中正常调用
完成声明后，在 C 代码中就可以像调用普通的 C 函数一样调用汇编函数了。

```c
// 在 os_core.c 中调用
void OS_Sched(void) {
    OS_CPU_SR  cpu_sr = 0;
    
    // 这里调用了汇编实现的函数
    // 编译器会生成一条 BL OS_CPU_SR_Save 指令
    // 并按照 AAPCS 规范，从 R0 寄存器获取返回值赋给 cpu_sr
    cpu_sr = OS_CPU_SR_Save(); 
    
    // ... 其他逻辑 ...
}
```

## 3. AAPCS：参数传递与返回值的“潜规则”

你可能会好奇：**C 语言是怎么把参数传给汇编的？汇编又是怎么把返回值交回给 C 语言的？**
答案是：它们都共同遵守了一份名为 **AAPCS**（ARM Architecture Procedure Call Standard，ARM 体系结构过程调用标准）的协议。

这是 ARM 架构下 C 与汇编协同编程的灵魂：

1. **参数传递（R0 ~ R3）**：
   - C 语言调用函数时，如果有参数，会把第 1 个参数放到 `R0` 寄存器，第 2 个放到 `R1`，依此类推直到 `R3`。
   - 如果参数超过 4 个，多出来的参数会被压入**栈（Stack）**中。
   - **在汇编中**：直接从 `R0`~`R3` 或栈中读取对应的值即可。

2. **返回值传递（R0）**：
   - 如果函数有返回值（且是 32 位及以下的整型/指针等），必须将返回值放入 `R0` 寄存器中。
   - C 语言在调用完函数后，会默认去 `R0` 里面取结果。
   - **例子**：在上面的 `OS_CPU_SR_Save` 汇编中，`MRS R0, PRIMASK` 就是把状态值放进了 `R0`，这就是它的“返回值”。

3. **寄存器保护（R4 ~ R11）**：
   - 汇编函数可以随意使用 `R0`~`R3` 和 `R12`（这叫“调用者保存寄存器”），不需要恢复它们的值。
   - 但是，如果汇编函数内部用到了 `R4`~`R11`，必须在函数开头把它们压栈保存（`PUSH`），并在返回前恢复（`POP`），因为 C 语言假设这些寄存器的值在函数调用前后不会被改变（这叫“被调用者保存寄存器”）。

## 4. 常见问题总结

1. **Undefined Symbol（未定义符号）**：
   - 原因：C 语言里调用了，但汇编里没有用 `EXPORT` 导出，或者汇编文件根本没有被加入到工程中参与编译。
2. **Implicit declaration of function（函数隐式声明）**：
   - 原因：汇编里写了，也导出了，但 C 语言没有在 `.h` 里声明就直接使用了。
3. **参数/返回值对不上**：
   - 原因：汇编里把返回值放在了 `R1`，但 C 语言去 `R0` 里找；或者 C 语言传了 3 个参数，汇编却去读栈。这就破坏了 AAPCS 标准。

理解了“符号解析”和“AAPCS 标准”，你就完全掌握了 C 语言和 ARM 汇编混合编程的核心密码！
