# uC/OS 临界区保护：三种 OS_CRITICAL_METHOD 详解

在实时操作系统（如 uC/OS-II/III）中，**临界区（Critical Section）**是指一段在执行期间**绝不能被中断打断**的代码区域。这些代码通常在操作操作系统的核心数据结构（如任务就绪表、TCB 链表等）。

为了保护临界区，uC/OS 提供了两个宏：
- `OS_ENTER_CRITICAL()`：进入临界区（通常是关闭中断）
- `OS_EXIT_CRITICAL()`：退出临界区（通常是恢复中断）

uC/OS 设计了 **3 种不同的实现方法（OS_CRITICAL_METHOD）**，以适应不同处理器架构和开发者的需求。

---

## 1. 方法 1 (OS_CRITICAL_METHOD == 1)
**最简单粗暴的方法：直接开关中断。**

### 实现原理
```c
#if OS_CRITICAL_METHOD == 1
#define OS_ENTER_CRITICAL()  asm("CPSID I")  // 直接关闭全局中断
#define OS_EXIT_CRITICAL()   asm("CPSIE I")  // 直接打开全局中断
#endif
```

### 优缺点分析
- **优点**：极其简单，代码执行最快，不需要保存任何状态。
- **致命缺点**：**不支持嵌套调用，会破坏中断原本的状态！**

**为什么危险？（嵌套问题场景）**
假设在调用某个函数**之前**，系统的中断**已经是关闭**的：
1. 此时中断是**关**的。
2. 调用某个使用了 `OS_ENTER_CRITICAL()` 的函数（执行了关中断指令，状态不变）。
3. 临界区代码执行完毕。
4. 调用 `OS_EXIT_CRITICAL()`（执行了开中断指令）。
5. **灾难发生**：原本函数外部希望中断是关闭的，但退出临界区时**被强制打开了**！这会引发严重的逻辑错误。

> **结论**：由于其固有的缺陷，这种方法在现代复杂的嵌入式系统中**几乎被完全弃用**。

---

## 2. 方法 2 (OS_CRITICAL_METHOD == 2)
**使用全局栈保存中断状态（通过 PUSH/POP）。**

### 实现原理
这种方法在进入临界区时，将当前处理器的状态寄存器（包含中断使能状态）压入**当前任务的堆栈**中；在退出临界区时，从堆栈中弹出状态寄存器的值。

```c
#if OS_CRITICAL_METHOD == 2
#define OS_ENTER_CRITICAL()  asm("PUSH {PRIMASK}"); asm("CPSID I") // 将中断状态压栈，然后关中断
#define OS_EXIT_CRITICAL()   asm("POP {PRIMASK}")                  // 从栈中恢复中断状态
#endif
```

### 优缺点分析
- **优点**：完美解决了嵌套调用的问题。无论进入临界区前中断是开还是关，退出时都会恢复到进入前的状态。
- **缺点**：
  1. **破坏了编译器的优化和预期**：大多数 C 编译器不喜欢在 C 代码中直接插入改变堆栈指针（SP）的汇编指令（PUSH/POP）。如果编译器在生成局部变量或进行函数调用时刚好依赖于 SP，这种强制压栈可能会导致栈错位，引发致命错误。
     
     **【致命错误案例演示】：**
     假设编译器在编译一个函数时，安排了如下的内存布局，它打算通过 `SP` 偏移量来读取局部变量：
     ```c
     void my_function(void) {
         int a = 10;
         int b = 20;
         
         OS_ENTER_CRITICAL(); // 展开为: asm("PUSH {PRIMASK}"); asm("CPSID I");
         
         // 此时 C 编译器不知道你悄悄 PUSH 了一个数据进栈！
         // 编译器原来以为变量 a 存放在 [SP + 0] 的位置，变量 b 在 [SP + 4]
         // 但因为你手动 PUSH 了 4 字节的 PRIMASK，真正的 SP 已经下移了！
         // 结果：下面的代码在读取 'a' 和 'b' 时，读到的全是错位的数据！
         int sum = a + b; 
         
         OS_EXIT_CRITICAL();  // 展开为: asm("POP {PRIMASK}");
     }
     ```
     *由于编译器无法感知内联汇编对 SP 的修改，它生成的取值指令 `LDR R0, [SP, #0]` 会直接取到你刚刚压进去的 `PRIMASK`，而不是变量 `a`，从而导致后续逻辑彻底崩溃！*

  2. **兼容性差**：不是所有的编译器（如早期的某些编译器）都允许内联汇编直接操作堆栈。

> **结论**：虽然解决了逻辑问题，但由于对编译器不够友好，存在栈破坏的风险，目前也**极少使用**。

## 3. 方法 3 (OS_CRITICAL_METHOD == 3)
**使用局部变量保存中断状态（最完美、最通用的方法）。**

### 实现原理
这是 uC/OS **官方最推荐，也是目前几乎所有移植方案（包括 STM32/Cortex-M）都在使用的方法**。
它要求在包含临界区的 C 函数中，**声明一个特定类型的局部变量（通常命名为 `cpu_sr`）**，用来临时保存中断状态。

在你的代码 [os_cpu.h](file:///d:/github/Quadcopter_2/lzh/usOSii/ucos_rebuild/UCOS_Port/os_cpu.h) 中正是使用了这种方法：
```c
#if OS_CRITICAL_METHOD == 3u
// 进入时：调用汇编函数获取当前状态保存到 cpu_sr，并在汇编里关闭中断
#define  OS_ENTER_CRITICAL()  {cpu_sr = OS_CPU_SR_Save();}
// 退出时：将保存在 cpu_sr 中的状态作为参数传给汇编函数，恢复中断状态
#define  OS_EXIT_CRITICAL()   {OS_CPU_SR_Restore(cpu_sr);}
#endif
```

配合你在 [os_cpu_a.s](file:///d:/github/Quadcopter_2/lzh/usOSii/ucos_rebuild/UCOS_Port/os_cpu_a.s) 中的汇编实现：
```assembly
OS_CPU_SR_Save
    MRS     R0, PRIMASK      ; 1. 读取当前中断屏蔽寄存器的状态到 R0 (作为 C 函数的返回值)
    CPSID   I                ; 2. 关闭全局中断
    BX      LR               ; 3. 返回

OS_CPU_SR_Restore
    MSR     PRIMASK, R0      ; 1. 把 R0 (即传进来的 cpu_sr 参数) 恢复到中断屏蔽寄存器中
    BX      LR               ; 2. 返回
```

### 为什么在函数里必须声明 `cpu_sr` 变量？
因为宏展开后变成了 `{cpu_sr = OS_CPU_SR_Save();}`。如果没有提前在函数的开头声明 `OS_CPU_SR cpu_sr;`，编译器就会报错找不到 `cpu_sr` 变量。
*注意：在 uC/OS-III 中，为了避免忘记声明，官方把这个变量声明也做成了一个宏 `CPU_SR_ALLOC();`。*

### 优缺点分析
- **优点**：
  1. **绝对安全**：完美支持临界区嵌套。
  2. **编译器友好**：不直接操作堆栈指针（SP），状态保存在 C 语言的局部变量（可能被编译器优化到寄存器里，或者存放在编译器分配好的安全栈空间里）。
  3. **架构无关**：具体的保存/恢复逻辑被封装到了独立的汇编函数中，C 代码部分完全跨平台。
- **缺点**：稍微麻烦一点点，要求开发者在每个使用临界区的函数开头手动声明一个局部变量 `cpu_sr`。

---

## 总结对比

| 方法 | 实现机制 | 支持嵌套? | 编译器友好? | 当前使用状态 |
| :---: | :--- | :---: | :---: | :--- |
| **1** | 直接开/关中断 | ❌ 否 | ✅ 是 | ⚠️ 已弃用（破坏中断状态） |
| **2** | 压入/弹出全局堆栈 | ✅ 是 | ❌ 否 | ⚠️ 极少使用（可能破坏 C 栈帧） |
| **3** | 保存到局部变量 `cpu_sr` | ✅ 是 | ✅ 是 | 👑 **绝对主流（官方推荐，STM32 标配）** |

通过使用**方法 3**，STM32 配合 Cortex-M 的 `PRIMASK` 寄存器，能够以最高效且最安全的方式保护 uC/OS 的内核数据！
