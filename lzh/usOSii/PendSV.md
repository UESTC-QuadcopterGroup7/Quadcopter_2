**PendSV (Pendable Service Call)** 是 ARM Cortex-M 系列内核（如 STM32 等）中的一种特殊异常（中断），在实时操作系统（RTOS，如 uC/OS-II、FreeRTOS）中扮演着**核心角色**。

它的主要用途是：**执行上下文切换 (Context Switching)**。

### 为什么需要 PendSV？
在没有 PendSV 之前，如果我们在 SysTick 中断（或其他高优先级中断）中直接进行上下文切换，可能会遇到一个严重问题：**中断嵌套冲突**。

1.  **场景**：假设有一个高优先级中断（比如 ADC 中断）正在运行。
2.  **SysTick 到来**：此时 SysTick 定时器中断触发，操作系统发现时间片到了，需要切换到另一个任务。
3.  **强制切换**：如果操作系统在 SysTick 中直接执行上下文切换，它会保存当前状态并加载新任务状态。但是，此时 CPU **还处于处理 ADC 中断的状态**（Usage Fault 或硬错误），因为中断还没完全退出。

**解决方案：PendSV**
PendSV 提供了一种**“延迟执行”**的机制：
*   **最低优先级**：PendSV 通常被配置为系统的**最低优先级**异常。
*   **挂起 (Pend)**：当操作系统（在 SysTick 或其他地方）决定需要切换任务时，它**不立即切换**，而是简单地将 PendSV 异常标志位置位（即“挂起” PendSV）。
*   **自动执行**：由于 PendSV 优先级最低，CPU 会先处理完所有其他高优先级中断（如 ADC、UART 等）。等到所有高优先级中断都处理完毕，系统空闲下来时，PendSV 异常才会触发。
*   **安全切换**：在 PendSV 的中断服务函数（Handler）中，执行真正的上下文切换操作。此时没有其他中断在运行，切换是安全的。

### PendSV 在 uC/OS-II 中的工作流程

1.  **触发**：
    *   **SysTick 中断**：调用 `OSTimeTick()` -> 发现需要调度 -> 调用 `OSIntCtxSw()`。
    *   **任务级调用**：任务调用 `OSTimeDly()` 或等待信号量 -> 发现需要调度 -> 调用 `OSCtxSw()`。

2.  **挂起 PendSV**：
    `OSIntCtxSw()` 或 `OSCtxSw()` 的实现通常很简单，就是向 NVIC（嵌套向量中断控制器）写入指令，触发 PendSV 异常。
    ```c
    // 典型的触发代码 (在 os_cpu.h 或 os_cpu_a.asm 中)
    NVIC_INT_CTRL_REG = NVIC_PENDSVSET_BIT; 
    ```

3.  **执行 PendSV_Handler** (在 `os_cpu_a.asm` 中实现)：
    这是真正的上下文切换发生的地方。
    *   保存当前任务的寄存器到堆栈（R4-R11，R0-R3等由硬件自动保存）。
    *   保存当前堆栈指针到 TCB。
    *   找到最高优先级的就绪任务。
    *   从新任务的 TCB 恢复堆栈指针。
    *   从新堆栈恢复寄存器（R4-R11）。
    *   执行异常返回（BX LR），硬件自动恢复剩余寄存器并跳转到新任务执行。

### 总结
**PendSV 是 RTOS 的“缓兵之计”。** 它允许操作系统在中断服务程序中“预约”一次上下文切换，但等到所有紧急的中断处理完之后再真正执行，从而保证了中断处理的实时性和系统稳定性。
